Да. Проект можно назвать условно **GitMessenger / GitGram / Macaroni Protocol**.

Суть:

**Мессенджер без собственного backend’а, где транспорт, хранение, история и синхронизация реализованы через обычный git-репозиторий.**

Одна из ключевых фич:

> Клиент можно развернуть как обычный HTML/JS-документ почти где угодно: на статическом хостинге, GitHub Pages, GitLab Pages, S3, nginx, локальном файле или флешке.

Backend всё ещё не нужен. Нужны только браузер, JavaScript и доступ к выбранному git-репозиторию через подходящий adapter.

Не “убийца Telegram”. Не “секьюрный мессенджер”. Не “платформа для миллионов”.

А честная штука:

> Мессенджер для небольших групп людей, где все сообщения являются файлами в git-репозитории, а клиент просто удобно это читает, пишет, индексирует и показывает.

## Главная философия

Проект не обещает приватность.

Наоборот:

> По умолчанию ваша переписка не приватна.
> Она лежит в git-репозитории.
> Кто имеет доступ к репозиторию — тот может её читать.
> Если репозиторий публичный — её может читать весь интернет.

Это не баг, это позиция.

Приватность — не часть базового протокола. Приватность может быть реализована поверх:

* PGP;
* age;
* пользовательские плагины;
* шифрование отдельных чатов;
* шифрование отдельных сообщений;
* шифрование вложений.

Базовая система решает только задачу:

> “Доставить текстовое сообщение от одного клиента другому через git”.

## Антицель

Проект **не пытается быть Telegram**.

Не нужно:

* realtime;
* звонки;
* видео;
* стикерпаки;
* сторис;
* каналы на миллион человек;
* модерация;
* удаление сообщений;
* самоуничтожающиеся сообщения;
* сложная криптография из коробки;
* идеальная доставка за миллисекунды.

Это скорее:

> ICQ/Jabber/почта, но поверх git, с нормальным интерфейсом на любой платформе, где можно крутить JavaScript.

## Целевая аудитория

Не “все люди”.

А:

* маленькие команды;
* семья;
* друзья;
* разработчики;
* параноидально-ироничные люди;
* люди, которым нужен простой async-чат;
* люди, которым нравится идея “backend — это git”.

Пример реального use case:

> “Мам, свари макарошки”
> Клиент создает JSON-файл, коммитит его, пушит в GitVerse.
> Клиент мамы раз в минуту делает fetch, видит новый файл, показывает сообщение.

   * GitVerse;

2. **JS-клиент**

3. **Репозиторий сообщений**

   * обычный git repo;
   * структура директорий;
   * JSON/YAML/Markdown-сообщения;

4. **Плагины**

   * шифрование;
   * форматирование;
   * вложения;
   * уведомления;
   * экспорт/импорт;
   * боты.

## Почему git

Потому что git уже умеет:

* хранить историю;
* синхронизировать;
* разрешать независимые изменения;
* работать оффлайн;
* делать pull/push;
* иметь пользователей;
* иметь доступы;
* хранить всё локально;
* давать аудит;
* работать через существующие хостинги.

Фактически git становится:

* базой данных;
* транспортом;
* журналом событий;
* системой репликации;
* системой бэкапа.

## Модель данных

Репозиторий примерно такой:

```text
repo/
  protocol.json

  users/
    alice.json
    bob.json
    mom.json

  chats/
    chat_01HXYZ/
      meta.json
      members.json

      messages/
        2026/
          06/
            08/
              2026-06-08T12-30-01.123Z_alice_a8f1c2.json
              2026-06-08T12-31-10.812Z_mom_9b4a11.json

      receipts/
        alice.json
        bob.json
        mom.json

  inbox/
    alice/
      2026-06-08T12-31-10.812Z_mom_9b4a11.json

    mom/
      2026-06-08T12-30-01.123Z_alice_a8f1c2.json
```

## Сообщение

```json
{
  "version": 1,
  "id": "2026-06-08T12-30-01.123Z_alice_a8f1c2",
  "chat_id": "chat_01HXYZ",
  "type": "text",
  "from": "alice",
  "to": ["mom"],
  "created_at": "2026-06-08T12:30:01.123Z",
  "text": "Мам, свари макарошки",
  "reply_to": null,
  "attachments": [],
  "meta": {
    "client": "Macaroni Messenger JS 0.1.0"
  },
  "signature": null
}
```

## Чаты

`meta.json`:

```json
{
  "id": "chat_01HXYZ",
  "title": "Семья",
  "created_at": "2026-06-08T12:00:00Z",
  "created_by": "alice",
  "visibility": "repo",
  "privacy": "public_by_design"
}
```

`members.json`:

```json
{
  "members": [
    {
      "id": "alice",
      "display_name": "Alice",
      "role": "owner"
    },
    {
      "id": "mom",
      "display_name": "Мама",
      "role": "member"
    }
  ]
}
```

## Доставка сообщений

Отправка:

1. Пользователь пишет сообщение.
2. Клиент создает JSON-файл.
3. Кладет его в `chats/<chat_id>/messages/...`.
4. Дополнительно кладет уведомление в `inbox/<recipient>/...`.
5. Делает commit.
6. Делает push.

Получение:

1. Клиент раз в N секунд/минут делает `git fetch`.
2. Проверяет новые коммиты.
3. Делает merge/rebase/pull.
4. Сканирует новые файлы.
5. Индексирует их в локальное хранилище через storage adapter.
6. Показывает уведомление.

## Почему `inbox`

Потому что сканировать всю историю каждый раз тупо.

`inbox/<user>` — это простая очередь уведомлений.

Сообщение всё равно лежит в чате, но inbox позволяет быстро понять:

> “Что нового лично для меня?”

Для 30 человек этого хватит с огромным запасом.

## Конфликты

Главный принцип:

> Сообщения никогда не редактируются на месте.

Каждое сообщение — новый файл с уникальным именем.

Имя:

```text
<timestamp>_<author>_<random/hash>.json
```

Например:

```text
2026-06-08T12-30-01.123Z_alice_a8f1c2.json
```

Два человека почти никогда не создадут один и тот же файл.

Даже если часы совпали — есть author + random/hash.

Поэтому классических git-конфликтов почти нет.

## Редактирование сообщений

В базовой версии — нет.

Или делается как новое событие:

```json
{
  "type": "edit",
  "target_message_id": "...",
  "new_text": "..."
}
```

Но старое сообщение остается в истории.

То есть “редактирование” — это не изменение файла, а событие поверх истории.

## Удаление сообщений

В базовой версии — нет.

Философия:

> Git помнит.
> Интернет помнит.
> Мы тоже помним.

Можно сделать событие:

```json
{
  "type": "delete_marker",
  "target_message_id": "..."
}
```

Клиент может скрыть сообщение визуально, но физически оно остается.

Это честнее, чем притворяться, что удаление в распределенной системе реально что-то гарантирует.

## Поиск

Поиск локальный.

Клиент при синхронизации складывает сообщения в локальный индекс:

```text
messages
  id
  chat_id
  from
  created_at
  text
  raw_json_path
```

И делает нормальный full-text search.

Git — источник истины.
Локальный индекс — runtime-specific cache: SQLite, IndexedDB, OPFS, in-memory для тестов или другой adapter.

Если индекс сломался — пересобирается из репозитория.

## Вложения

Базово — ссылками.

```json
"attachments": [
  {
    "type": "url",
    "url": "https://example.com/file.zip",
    "title": "Файл"
  }
]
```

Почему не хранить вложения в git?

Потому что git не любит большие бинарники.

Опционально:

* Git LFS;
* S3;
* WebDAV;
* Synology share;
* локальный file server;
* IPFS;
* просто URL.

Но MVP — только ссылки.

## Уведомления

Без push-сервера.

Просто polling:

* каждые 60 секунд;
* либо каждые 15 секунд в активном окне;
* либо вручную “обновить”.

Для “мам, свари макарошки” этого достаточно.

Можно оптимизировать:

* `git fetch --depth`;
* проверять только remote HEAD;
* использовать Git hosting API для проверки новых коммитов;
* webhook в будущем.

Но MVP — тупо fetch.

## Регистрация

Самая прекрасная часть.

Пользователь не регистрируется в мессенджере.

Он регистрируется в git-хостинге.

Например:

> Чтобы пользоваться мессенджером, создайте аккаунт в GitVerse и personal access token.

Клиент просит:

* URL репозитория;
* логин;
* токен;
* имя пользователя.

Всё.

Мессенджер не хранит пользователей на своем сервере, потому что сервера нет.

## Оператор данных

Сатирическая позиция проекта:

> Мы не являемся оператором ваших сообщений.
> Мы не храним ваши сообщения.
> Мы не обрабатываем ваши сообщения.
> Мы предоставляем клиент, который пишет файлы в выбранный вами git-репозиторий.
> Где вы разместили репозиторий — туда и вопросы.

Это, конечно, надо формулировать аккуратно юридически, но продуктовая сатира понятна.

## Безопасность

Базовая модель:

* сообщения открытые;
* приватность не обещается;
* доступ определяется настройками репозитория;
* если репозиторий публичный — всё публично;
* если приватный — всё доступно тем, у кого есть доступ;
* git-хостинг потенциально видит всё.

Честный баннер при первом запуске:

> Этот мессенджер не является приватным.
> Не отправляйте сюда секреты, пароли, интимные фото, коммерческую тайну и всё, что не хотите увидеть в интернете.

## Шифрование как плагин

Плагин может перехватывать отправку:

До:

```json
"text": "Мам, свари макарошки"
```

После:

```json
"type": "encrypted",
"encryption": {
  "plugin": "pgp",
  "recipients": ["mom_key_id"]
},
"payload": "-----BEGIN PGP MESSAGE-----..."
```

Базовый клиент без плагина покажет:

> Это зашифрованное сообщение. Установите плагин PGP.

То есть шифрование не ломает протокол.

## Плагины

Плагины могут делать:

* PGP;
* Markdown rendering;
* emoji packs;
* link preview;
* вложения;
* импорт из Telegram;
* экспорт в HTML;
* ботов;
* автоперевод;
* фильтры;
* локальную модерацию;
* темы оформления.

Главное правило:

> Ядро тупое. Всё сложное — в плагины.

## Боты

Бот — это просто клиент, который тоже читает repo и пишет сообщения.

Например:

* RSS-бот;
* weather-бот;
* CI-бот;
* “напоминалка”;
* бот, который пишет “мам, макароны готовы”.

Бот может крутиться:

* на Mac Mini;
* на VDS;
* на GitHub Actions;
* на GitLab CI;
* где угодно.

## Статусы прочтения

Не обязательны.

Но можно сделать:

```text
chats/<chat_id>/receipts/alice.json
```

```json
{
  "last_seen_message_id": "...",
  "last_seen_at": "2026-06-08T12:40:00Z"
}
```

Это будет отдельный файл, который клиент обновляет.

Но тут возможны конфликты, потому что файл один.

Лучше append-only:

```text
receipts/alice/2026-06-08T12-40-00Z.json
```

И клиент берет последний.

## Онлайн-статус

Не нужен.

Можно сделать “last sync”, но это уже лишнее.

Проект не про “онлайн/оффлайн”, а про асинхронные сообщения.

## Производительность

Для 30 человек и текстовых сообщений git выдержит вообще спокойно.

Даже если:

* 30 человек;
* 100 сообщений в день каждый;
* 3000 сообщений в день;
* 1 KB на сообщение;

это 3 MB JSON в день.

За год — около 1 GB сырого текста, и то с огромным запасом.

Реально будет меньше.

Проблемы начнутся не от сообщений, а если начать пихать бинарники.

Поэтому вложения — ссылками.

## UX

Пользователь должен видеть не git, а чат.

Интерфейс:

* список чатов слева;
* сообщения справа;
* поле ввода снизу;
* статус синхронизации;
* кнопка “обновить”;
* индикатор “последний fetch 42 секунды назад”.

При ошибке push:

> Сообщение сохранено локально. Не удалось отправить. Повторим позже.

При конфликте:

> Синхронизация задерживается. Клиент пытается автоматически слить изменения.

Никаких слов “merge conflict” для мамы.

## Локальная очередь исходящих

Если нет интернета:

1. Сообщения пишутся в локальную очередь.
2. UI показывает “ожидает отправки”.
3. Когда интернет появился — клиент делает commit/push.

То есть оффлайн работает естественно.

## Коммиты

Можно делать один commit на сообщение:

```text
message: alice -> mom: 2026-06-08T12:30:01Z
```

Но лучше батчить:

* каждые 10 секунд;
* или при потере фокуса;
* или сразу для простоты MVP.

MVP — commit на сообщение.
Потом оптимизировать.

## Ветки

Лучше не усложнять.

Одна ветка:

```text
main
```

Все пишут в main.

Потому что сообщения — уникальные файлы, конфликтов почти нет.

Для продвинутого режима можно:

* branch per user;
* merge bot;
* pull requests как сообщения, лол.

Но это уже красноглазие.

## README-позиционирование

Пример:

```text
Macaroni Messenger is a git-backed messenger for small groups on any platform that can run JavaScript.

It has no central server, no proprietary backend, no default encryption,
```

## MVP

Первая версия должна уметь только:

1. Создать локальный профиль.
2. Подключить git repo.
3. Создать чат.
4. Добавить участника по username.
5. Отправить текстовое сообщение.
6. Получить текстовое сообщение через polling.
7. Показать историю.
8. Сделать локальный поиск.
9. Работать оффлайн.
10. Переживать restart.

Всё.

Никаких вложений, плагинов, PGP, read receipts в первой версии.

## Стек

Главное ограничение:

> Мессенджер должен работать на любой платформе, где можно крутить JavaScript.

Первоклассный target:

> Статический HTML/JS-клиент, который можно разместить где угодно.

Поэтому ядро не должно быть привязано к Electron, Tauri, конкретному UI-фреймворку или конкретной базе.

Базовый стек:

* TypeScript;
* framework-agnostic core;
* JSON protocol validators;
* git transport adapter;
* local index storage adapter;
* UI adapters под конкретные платформы.

Возможные runtime targets:

* static HTML/JS app;
* Browser;
* Node.js;
* Electron;
* Tauri/WebView;
* mobile WebView;
* серверный bot/client runtime.

Для первого MVP можно взять самый быстрый runnable target, но архитектурно это не "Electron-мессенджер". Это JS-first messenger, где Electron или браузер - только оболочка.

## Риски

Главные проблемы:

1. Git-хостинг может не любить частые push/fetch.
2. Токены пользователей надо хранить аккуратно.
3. Маме нельзя показывать git-ошибки.
4. Публичность надо объяснять максимально жирно.
5. Репозиторий может разрастись.
6. Вложения нельзя хранить как попало.
7. Клиент должен быть тупоустойчивым.

## Почему это может реально взлететь как пет-проект

Потому что идея:

* смешная;
* простая;
* объясняется за 30 секунд;
* хорошо ложится в open source;
* не требует серверов;
* троллит современную регуляторную и privacy-риторику;
* при этом реально работает для маленьких групп.

Это не единорог.

Это идеальный проект категории:

> “лол, я сделал мессенджер поверх git, и теперь моя мама пишет мне через коммиты”.

И самое опасное — он реально реализуем.
