# Main Project Principle

**Do not make things complex where they can be funny.**

That does not change the fact that this is a full application.

Modern development often starts with the question:

> How do we do this correctly?

Macaroni Messenger starts with another question:

> How do we make this as simple as possible without breaking common sense?

If there are two solutions:

### Option A

- Kubernetes
- PostgreSQL
- Redis
- OAuth
- WebSocket Gateway
- Service Discovery

### Option B

- HTML
- Git
- JSON

Macaroni Messenger chooses option B by default.

We do not believe every task must be solved with complexity.

We believe many modern applications are overcomplicated into a religion.

So Macaroni Messenger is built on a few radical principles:

- If data can be stored in files, store it in files.
- If Git already stores history, use Git.
- If an application can be built as one HTML file, build it as one HTML file.
- If a problem appears after ten years of usage, that is a problem for ten years from now.
- If a chat becomes too large, create a new repository.

Macaroni Messenger is not a joke.

It simply refuses to complicate solutions without a reason.

That is why some technical decisions may look like jokes.

Sometimes they are jokes.

More often, they are just the simplest working option.

We do not promise perfect architecture.

We promise understandable architecture.

We do not promise infinite scalability.

We promise that the user understands where their messages are stored.

We do not promise magic.

We promise that everything can be explained in five minutes.

If, after reading the architecture, you think:

> "Wait, is that really all?"

Then the project is doing its job.

# Why Macaroni Messenger Exists

Macaroni Messenger is not a political statement.

Macaroni Messenger is not a manifesto.

Macaroni Messenger is not trying to change the world.

It came from a much simpler thought.

Every year, sending a simple message becomes more complicated.

Services block.

Services forbid.

Services require registration.

Services require a phone number.

Services require an app.

Services require app updates.

Services require operating system updates.

Services require endless rules, demands, restrictions, and procedures.

Services require a new expensive phone just to send an eight-bit emoji.

Services require paid subscriptions and show ads.

At some point, the obvious question appears:

> Why do I need infrastructure the size of a small bank to text my mom "boil some macaroni"?

Macaroni Messenger starts from a different idea.

To send a message, you need only three things:

- text;
- a way to store it;
- a way to deliver it.

Everything else is additional functionality.

Sometimes useful.

Sometimes not.

We are not trying to fight laws.

We are not trying to bypass restrictions.

We are not trying to defeat governments, corporations, or the industry.

We simply believe that sending a message to another person should not require complex infrastructure.

Also, it is genuinely a funny way to approach the problem.

That is why Macaroni Messenger uses:

- HTML;
- Git;
- JSON.

Not because it is modern.

Not because it is fashionable.

Not because it is correct.

Because it is enough.

If all modern messengers disappear tomorrow, Macaroni Messenger can still work.

Its foundation is made of technologies that have existed for decades and are everywhere.

The project is based on one simple idea:

> Sending a message to another person should be simpler than the system of restrictions built around that ability.

If one HTML file is enough, then one HTML file is enough.

# On Side Effects

Macaroni Messenger did not appear because the world is not regulated enough.

It also did not appear because the world is regulated too much.

It appeared because people too often try to solve a local problem with a global tool.

Engineering has a simple rule.

Every intervention has side effects.

Sometimes the side effects cost more than the original problem.

When solving one problem breaks ten other systems, the obvious question is:

Maybe the problem was not the technology.

Maybe the problem was the chosen method.

Macaroni Messenger is not an answer to one specific rule, law, or organization.

It is a reminder that communication can be much simpler.

If sending a message requires:

- a browser;
- an HTML file;
- a git repository;

then an interesting question appears.

What exactly needs to be restricted in this scheme?

HTML?

Git?

Text files?

The project is not trying to fight restrictions.

The project simply demonstrates how hard it is to restrict a system built almost entirely from the basic building blocks of the internet.

Macaroni Messenger exists because sometimes the most resilient solution is also the simplest and funniest one.
